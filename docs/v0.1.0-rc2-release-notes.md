## Summary

v0.1.0-rc2 is a safety-focused release that makes vxlan-tracer safe to run on
hosts with pre-existing CNI or operator-managed TC programs (Cilium, Calico, Flannel,
tc flower, etc.). It also adds interface discovery and diagnostic bundle collection
subcommands, and corrects several technically overconfident claims in documentation
and outreach drafts.

---

## Breaking changes from rc1

**TC filter behavior changed.** vxlan-tracer now attaches at priority 50000 / handle
0x7674_0001 (instead of priority 1 / handle 0x0001). This means:

- A host that had rc1's priority-1 TC filter still attached (from an interrupted run)
  will not automatically clean up that old filter when rc2 runs. Remove it manually:
  ```
  tc filter del dev <underlay> ingress prio 1
  tc filter del dev <overlay>  egress  prio 1
  ```
  Then run rc2 normally.

- rc2 and rc1 cannot run simultaneously on the same host (rc2's concurrent-run lock
  does not prevent rc1's lock-free attachment from running alongside it).

## New in rc2

### Ownership-safe TC attachment

vxlan-tracer now exclusively uses TC priority 50000 and handle major 0x7674 ('vt' in
ASCII). It never touches TC filters at any other priority or handle. Specifically:

- Cilium's priority-1 filters (cil_from_netdev, cil_to_netdev) are not touched.
- Calico's priority-1 filters are not touched.
- Operator-created tc flower rules at any priority other than 50000 are not touched.
- A filter at priority 50000 with a DIFFERENT handle is not deleted (collision handling).

### Deterministic cleanup

All resources created by vxlan-tracer are removed on normal exit (end of `--duration`
window) and on SIGINT / SIGTERM:

- TC filters at priority 50000 / handle 0x7674_0001
- clsact qdiscs (only if vxlan-tracer created them and no other filters remain)
- Pinned BPF maps under `--pin-dir`
- Pin directory (if now empty)

**SIGKILL exception:** If the process is killed with `kill -9`, TC filters remain
attached. Remove them manually:
```bash
tc filter del dev <underlay> ingress prio 50000
tc filter del dev <overlay>  egress  prio 50000
```

### Concurrent-run protection

Only one vxlan-tracer invocation may run at a time. A second run will fail immediately
with a clear error message and not disturb the first run's TC filters or maps.

### `vxlan-tracer interfaces` subcommand

Enumerates VXLAN-type interfaces on the host with their VNI, VXLAN port, MTU, and
inferred underlay. No root or BPF privileges required.

```bash
$ vxlan-tracer interfaces
VXLAN interfaces on this host:

  NAME          VNI    PORT    MTU     UNDERLAY
  flannel.1     1      8472    1450    eth0

Suggested invocations:
  sudo vxlan-tracer --overlay flannel.1 --underlay eth0
```

Notes:
- Enumerates VXLAN overlay interfaces only. Does not enumerate pod veth interfaces.
- The UNDERLAY column is an inference from the VtepDevIndex rtnetlink field.
  When no explicit VTEP device is configured (common in Flannel), it shows `(unknown)`.
  Treat as a suggestion, not an authoritative configuration value.
- JSON output via `--json`.

### `vxlan-tracer collect-environment` subcommand

Creates a privacy-safe tar.gz diagnostic bundle for attaching to a GitHub issue.
(`collect-support` is a deprecated alias; both names work.)

```bash
$ vxlan-tracer collect-environment
```

Included: kernel version, VXLAN interface table, BTF vmlinux status, BPF filesystem
mounts, kprobe symbol availability indicator, tool version string.

Not included: IP or MAC addresses, route tables, firewall rules, credentials, packet
captures, process lists.

`--dry-run` flag prints what would be collected without creating any file.

### `--keep-state` flag

`vxlan-tracer --keep-state` skips TC filter and map cleanup on exit. Intended for lab
and debugging use only. Do not use on shared or production hosts. Do not use in demos
or external-run instructions.

## Corrections in rc2

### Verdict count: six, not five

The tool has always had six verdict constants. Documentation and outreach drafts
incorrectly described five verdicts. The correct six are:

1. `NO_ISSUE_OBSERVED`
2. `PTB_DELIVERED`
3. `PTB_SUPPRESSED`
4. `VXLAN_FRAGMENTATION_OBSERVED`
5. `VXLAN_MTU_RISK`
6. `VXLAN_MTU_MISCONFIGURATION`

### tcpdump and PTBs

Earlier documentation incorrectly stated that tcpdump/AF_PACKET cannot see incoming
PTBs. This is wrong. AF_PACKET captures at the physical layer, before netfilter, so
a correctly placed `tcpdump -i <underlay> "icmp[0]==3 and icmp[1]==4"` CAN see PTBs
even if they are subsequently dropped by an iptables INPUT rule.

vxlan-tracer's advantage over a manual tcpdump workflow is automated simultaneous
measurement at two observation points (pre-netfilter and post-netfilter), without
requiring two concurrent capture sessions.

### Cloud/network fragmentation claims

Corrected "fragmented VXLAN UDP is commonly dropped silently by cloud fabric" to:
"fragmented IP packets may be dropped or mishandled by networks and middleboxes."
The previous claim was overconfident and not uniformly true.

### PTB localization claims

Clarified that `PTB_SUPPRESSED` means the signal is "consistent with a drop between
the TC hook and icmp_rcv." It does not identify which specific firewall rule or eBPF
code path is responsible.

### vxlan-tracer interfaces scope

Removed claims that `interfaces` enumerates pod veth MTUs. It does not. It enumerates
VXLAN overlay interfaces (e.g., vxlan0, flannel.1, vxlan.calico, cilium_vxlan).

### GSO/GRO qualification

The `VXLAN_MTU_RISK` verdict may be a false positive on hosts with GSO enabled on
the overlay interface. See `docs/gso-gro-limitations.md`. The `VXLAN_FRAGMENTATION_OBSERVED`
verdict (via ip_do_fragment) is not affected by GSO.

## Known limitations (unchanged from rc1)

- `ip_do_fragment` is a global kernel function. Fragmentation from non-VXLAN traffic
  (IPsec, GRE, other tunnels) is included in the count.
- Per-VNI or per-pod attribution is not possible with current BPF helpers.
- The tool has been validated in a netns lab on four kernels (5.15â€“6.10, aarch64 and
  x86_64). Real CNI environments (Flannel, Calico, Cilium) have not been tested.
- SIGKILL leaves TC filters attached (see above for manual cleanup).
- GSO super-packets may cause false `VXLAN_MTU_RISK` verdicts (see above).

---

## Not in rc2

- Per-pod or per-namespace VXLAN attribution
- GSO detection or qualification at the TC egress hook
- Automated tracer run inside collect-environment
- Telemetry (see docs/telemetry-proposal.md for the proposal; no implementation)
- Any outreach, posting, or public announcement

## Upgrade from rc1

1. Stop any running rc1 instance (Ctrl-C or SIGTERM).
2. Verify rc1's filters are gone: `tc filter show dev <underlay> ingress prio 1`
   If they remain: `tc filter del dev <underlay> ingress prio 1` and similarly for
   overlay egress.
3. Replace the binary with the rc2 build.
4. Run normally. rc2 will use priority 50000 and clean up on exit.
