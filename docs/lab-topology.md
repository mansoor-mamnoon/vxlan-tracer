# Lab Topology

The lab uses Linux network namespaces to simulate a two-node VXLAN cluster on
a single host. No external hardware is required.

## Topology diagram

```
ns1 (sender / tracer host)          ns2 (receiver / "remote pod")
──────────────────────────          ──────────────────────────────
vxlan0: 10.244.0.1/24               vxlan0: 10.244.0.2/24
MTU: 1500 (intentionally wrong)     MTU: 1500
        │                                   │
veth1: 192.168.100.1/24             veth2: 192.168.100.2/24
MTU: 1500                           MTU: 1500
        └──────────── veth pair ────────────┘
```

## VXLAN overhead math

The kernel's MTU check operates at the **IP layer**: it compares the outer IP
packet length against the underlay interface MTU (1500). The outer Ethernet
header (14 bytes) is NOT part of this comparison.

```
IP-layer overhead (inner IP → outer IP):
  inner ETH  = 14 bytes  (VXLAN carries the full inner Ethernet frame)
  outer IP   = 20 bytes
  outer UDP  =  8 bytes
  VXLAN hdr  =  8 bytes
              ─────────
              50 bytes

For inner IP = 1500 bytes (TCP MSS=1460):
  outer IP packet = 1500 + 50 = 1550 bytes
  veth1 MTU       = 1500 bytes
  IP-layer excess = 1550 - 1500 = 50 bytes  ← kernel triggers fragment/drop here

  wire frame on wire = 14 (outer ETH) + 1550 = 1564 bytes  (informational only)
```

The intentional wrong MTU (1500 on vxlan0) triggers the VXLAN blackhole
scenario. The correct MTU would be 1450 (= 1500 − 50), which keeps the outer
IP packet at exactly 1500 bytes (1450 + 50 = 1500).

## Expected behavior

After `lab-up`:

- `ip netns exec ns1 ping -c 3 -s 56 10.244.0.2` — should succeed (56-byte
  ICMP payload → inner IP 84 bytes → outer IP 134 bytes, well under 1500)
- `ip netns exec ns1 ping -c 3 -s 1400 10.244.0.2` — may succeed (inner IP
  1428 bytes → outer IP 1478 bytes, still under 1500)
- `ip netns exec ns1 ping -c 3 -s 1452 10.244.0.2` — should fragment or fail
  (inner IP 1480 bytes → outer IP 1530 bytes, 30 bytes over underlay MTU 1500)
- `ip netns exec ns1 curl http://10.244.0.2/large.bin` — expected to stall
  after ~8KB (TCP MSS 1460 → inner IP 1500 → outer IP 1550, 50 bytes over MTU)

**Note on local netns topologies:** In this two-namespace veth setup, DF=0
outer fragments may actually reassemble successfully in ns2 (there is no
middlebox to drop them). The blackhole only reproduces as a hard stall if:
- df=set is configured on vxlan0 (forcing DF=1 → outer drops → PTBs generated)
- OR a realistic middlebox (iptables DROP rule or similar) drops the fragments

Without one of these, the lab may show successful large transfers despite
fragmentation events. The `ip_do_fragment` kprobe WILL still fire (fragmentation
events are real), but end-to-end connectivity may not break. This is documented
in evidence/day-01.md and is a known limitation of the local-netns lab.

## HTTP server in ns2

The setup script starts a Python HTTP server in ns2 serving files from
`/tmp/vxlan-lab-www/`. The large file (`large.bin`, 5MB of urandom) and small
file (`small.txt`) are created at setup time.

## bpftrace probes (Linux-only)

The bpftrace probes in `spikes/bpftrace/` run in the host network namespace
but observe traffic generated by ns1. Attach them before running smoke tests:

```sh
# Terminal 1 (host):
sudo bpftrace spikes/bpftrace/ip_do_fragment.bt

# Terminal 2:
ip netns exec ns1 curl --max-time 10 http://10.244.0.2/large.bin -o /dev/null
```
