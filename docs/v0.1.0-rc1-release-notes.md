# vxlan-tracer v0.1.0-rc1

Release candidate 1 for v0.1.0.

---

## What vxlan-tracer diagnoses

vxlan-tracer is a Linux diagnostic tool that identifies VXLAN MTU misconfiguration
and fragmentation blackholes using eBPF.

**Symptom:** small requests (ping, small HTTP) work; large requests (file transfers,
`kubectl cp`, large API payloads) silently stall, fragment, or drop after a few
kilobytes. No error is logged.

The tool attaches TC sched_cls hooks and kprobes to observe VXLAN traffic and
produce a structured verdict. It runs for a configurable duration, observes
existing traffic passively, and exits with the result. It is not a daemon and does
not produce continuous metrics.

It diagnoses five conditions:

| Verdict | Meaning |
|---------|---------|
| `VXLAN_MTU_MISCONFIGURATION` | overlay MTU > underlay MTU − 50 (wrong configuration) |
| `VXLAN_FRAGMENTATION_OBSERVED` | ip_do_fragment invocations corroborated by oversized TC-egress packets |
| `VXLAN_MTU_RISK` | MTU margin is too small but no fragmentation seen yet |
| `PTB_DELIVERED` | ICMP "Packet Too Big" entered the kernel post-netfilter (`icmp_rcv`) |
| `PTB_SUPPRESSED` | PTBs counted at TC ingress but dropped before `icmp_rcv` — suppressed by netfilter |

Output is machine-readable JSON (`--json`) or human-readable text.

## Supported architectures and VXLAN ports

| Architecture | Status |
|-------------|--------|
| Linux amd64 (x86_64) | Supported; BPF objects compiled with `__TARGET_ARCH_x86` |
| Linux arm64 (aarch64) | Supported; BPF objects compiled with `__TARGET_ARCH_arm64` |
| macOS | Not supported (Linux only) |

BPF kprobe objects (`kprobes.bpf.o`, `frag_kprobes.bpf.o`) are architecture-specific
and are **not interchangeable** between amd64 and arm64 archives.

VXLAN UDP ports 4789 (default) and 8472 (Flannel/Calico) are both supported.
The port is auto-detected from the overlay interface at startup via rtnetlink,
or can be specified with `--vxlan-port`.

| 6.8.0-1052-azure | x86_64 | 5/5 PASS | GitHub Actions (tested before scenario 6 was added) |

CO-RE (`CONFIG_DEBUG_INFO_BTF`) is required. `/sys/kernel/btf/vmlinux` must be
present. Portability to untested kernels is likely but not guaranteed; validate on
your target kernel before production use.

The `ip_do_fragment` and `icmp_rcv` symbols must be present as `T` symbols in
`/proc/kallsyms`. Both were kprobeable on all tested kernels.

## Quick start from packaged archive

Download `vxlan-tracer-linux-amd64.tar.gz` (or `arm64`) from the release assets
and verify the checksum:

```bash
sha256sum -c checksums.sha256
```

Extract and run the demo (Linux only, requires root):

```bash
tar -xzf vxlan-tracer-linux-amd64.tar.gz
cd vxlan-tracer-linux-amd64
sudo bash scripts/preflight.sh        # verify kernel/BTF prerequisites
sudo bash scripts/demo.sh             # ~25 s stale-MTU demo
```

Check version:

```bash
./vxlan-tracer --version
./vxlan-tracer --help
```

Run the full diagnostic:

```bash
sudo ./vxlan-tracer --overlay vxlan0 --underlay eth0 --duration 30s
sudo ./vxlan-tracer --overlay flannel.1 --underlay eth0 --json
```

## What the packaged demo produces

The demo creates a two-netns VXLAN topology with a stale overlay MTU (1450),
runs five oversized pings across the VXLAN, and reports a verdict and MTU
recommendation. It cleans up all network state on exit.

Expected human-readable output:

```
Verdict:  VXLAN_FRAGMENTATION_OBSERVED
Evidence:
  ip_do_fragment events:   5
  largest outer IP seen:   1438 B
  underlay MTU:            1400 B  (outer packet exceeded by 38 B)
Recommendation:
  set overlay MTU to 1350 B or lower
  (VXLAN overhead is 50 B; safe overlay MTU = underlay MTU − 50)
Scope:
  global fragmentation counter corroborated by VXLAN TC egress
  (both ip_do_fragment and oversized outer packets observed)
```

Expected JSON:

```json
{
  "verdict": "VXLAN_FRAGMENTATION_OBSERVED",
  "fragmentation_scope": "global_corroborated",
  "overlay": "vxlan0",
  "underlay": "demo-veth1",
  "overlay_mtu": 1450,
  "underlay_mtu": 1400,
  "recommended_overlay_mtu": 1350,
  "frag_events_total": 5,
  "max_outer_ip_len": 1438
}
```

## Known limitations

**Fragmentation scoping:** `ip_do_fragment` is a global kernel function.
`frag_events_total` counts all IP fragmentation on the host, not only VXLAN
fragmentation. When both `ip_do_fragment` invocations and oversized TC egress
packets are observed during the same run, the verdict is `global_corroborated`;
the message makes the limitation explicit.

This does not confirm that fragmented VXLAN packets are actually dropped. Cloud
fabric commonly drops fragmented UDP silently, but this tool cannot confirm loss —
it only observes fragmentation events.

**PTB inner 5-tuple:** vxlan-tracer cannot extract the inner 5-tuple from ICMP PTB
payloads. PTB counts reflect all Type 3 Code 4 messages on the underlay interface,
not per-flow counts.

**Single interface pair:** The tool monitors one overlay and one underlay interface
pair per invocation.

**IPv4 only:** IPv6 underlay (ICMPv6 Type 2) is not supported.

**No active probing:** vxlan-tracer observes existing traffic passively; it does not
generate probe packets.

**bpf_get_netns_cookie:** Network namespace-scoped filtering is not available for
TC sched_cls and kprobe program types on tested kernels. The tool operates at
global kernel scope.

**ARM64 live demo:** The end-to-end demo was executed from the amd64 packaged
archive only. The arm64 archive passed all structural and scenario qualification
gates; a live demo from the arm64 archive was not run.

## What is not yet complete

**Real Kubernetes/CNI validation is not done.** The six diagnostic scenarios
(including port 8472) were validated in a Linux network-namespace lab environment,
not on a real k3s or flannel cluster. No cross-node pod traffic was observed. The
tool has not been run against actual CNI overlay traffic.

Real two-node Kubernetes/CNI validation is a V1 milestone. Do not interpret the
port 8472 support or scenario results as proof of flannel compatibility.

**No cloud deployment testing.** The fragmentation-observed verdict was confirmed in
a netns lab. Whether ip_do_fragment fires during actual cloud VXLAN traffic, and
whether cloud fabric silently drops the resulting fragments, has not been tested.

## v0.1.0 vs V1

v0.1.0-rc1 is a lab-validated prototype. It is not production-ready and makes no
claim of production correctness, all-kernel compatibility, or Kubernetes validation.

V1 targets: real Kubernetes/CNI validation, per-VNI attribution, rolling-window
comparison for PTB suppression, additional kernel coverage.
