# vxlan-tracer v0.1.0-rc1 release notes

Release candidate 1 for v0.1.0. Not yet tagged. Full qualification evidence
is in `evidence/`.

---

## What vxlan-tracer diagnoses

vxlan-tracer is a Linux diagnostic tool that identifies VXLAN MTU misconfiguration
and fragmentation blackholes using eBPF. It attaches TC sched_cls hooks and kprobes
to observe VXLAN traffic and produce a structured verdict.

It diagnoses five conditions:

| Verdict | Meaning |
|---------|---------|
| `VXLAN_MTU_MISCONFIGURATION` | overlay MTU > underlay MTU − 50 (wrong configuration) |
| `VXLAN_FRAGMENTATION_OBSERVED` | ip_do_fragment invocations corroborated by oversized TC-egress packets |
| `VXLAN_MTU_RISK` | MTU margin is too small but no fragmentation seen yet |
| `PTB_DELIVERED` | ICMP "Packet Too Big" entered the kernel post-netfilter (icmp_rcv) |
| `PTB_SUPPRESSED` | PTBs counted at TC ingress but dropped before icmp_rcv — suppressed by netfilter |

Output is machine-readable JSON (`--json`) or human-readable text. The tool runs
for a configurable duration, observes traffic, and exits with the verdict. It does
not run as a daemon and does not produce continuous metrics.

## Supported architectures and VXLAN ports

| Architecture | Status |
|-------------|--------|
| Linux amd64 (x86_64) | Supported; BPF objects compiled with `__TARGET_ARCH_x86` |
| Linux arm64 (aarch64) | Supported; BPF objects compiled with `__TARGET_ARCH_arm64` |
| macOS | Not supported (Linux only) |

BPF kprobe objects (`kprobes.bpf.o`, `frag_kprobes.bpf.o`) are architecture-specific
and are **not interchangeable** between amd64 and arm64 archives.

VXLAN UDP ports 4789 (default) and 8472 (Flannel/Calico) are both supported.
The port is auto-detected from the overlay interface at startup via rtnetlink,
or can be specified with `--vxlan-port`.

CO-RE (`CONFIG_DEBUG_INFO_BTF`) is required. BTF (`/sys/kernel/btf/vmlinux`) was
present on all tested kernels. Portability to untested kernels is likely but not
guaranteed; test on your target kernel before production use.

The `ip_do_fragment` and `icmp_rcv` symbols are required for fragmentation and PTB
detection, respectively. Both were kprobeable (`T` symbol in `/proc/kallsyms`) on
all tested kernels.

## Quick demo

From the extracted archive on a Linux host with root access:

```bash
tar -xzf vxlan-tracer-linux-amd64.tar.gz
cd vxlan-tracer-linux-amd64
sudo bash scripts/demo.sh
```

The demo creates a two-netns VXLAN topology with a stale overlay MTU, runs five
oversized pings, and reports `VXLAN_FRAGMENTATION_OBSERVED` with a recommended MTU.
It cleans up all network state on exit.

The packaged demo was run twice end-to-end on Linux (x86_64, kernel 6.8.0-1059-azure,
CI run 27887911218). Both runs produced `VXLAN_FRAGMENTATION_OBSERVED` with
`fragmentation_scope=global_corroborated`. Cleanup was verified after each run.

## Known limitations

**Fragmentation scoping:** `ip_do_fragment` is a global kernel function. `frag_events_total`
counts all IP fragmentation on the host, not only VXLAN fragmentation. When both
`ip_do_fragment` invocations and oversized TC egress packets are observed during the
same run, the verdict is `global_corroborated`; the message makes the limitation explicit.
This does not confirm that fragmented VXLAN packets are actually dropped. Cloud fabric
commonly drops fragmented UDP silently, but this tool cannot confirm loss — it only
observes fragmentation events.

**PTB inner 5-tuple:** vxlan-tracer cannot extract the inner 5-tuple from ICMP PTB
payloads. PTB counts reflect all Type 3 Code 4 messages observed on the underlay
interface, not per-flow counts.

**Single interface pair:** The tool monitors one overlay and one underlay interface
pair per invocation.

**IPv4 only:** IPv6 underlay (ICMPv6 Type 2) is not supported.

**No active probing:** vxlan-tracer observes existing traffic passively; it does not
generate probe packets.

**bpf_get_netns_cookie:** Network namespace-scoped filtering via `bpf_get_netns_cookie`
is not available for TC sched_cls and kprobe program types on tested kernels. The
tool operates at global kernel scope.

## What is not yet complete

**Real Kubernetes/CNI validation is not done.** The six diagnostic scenarios (including
port 8472) were validated in a Linux network-namespace lab environment, not on a real
k3s or flannel cluster. No cross-node pod traffic was observed. The tool has not been
run against actual CNI overlay traffic.

This is not an rc1 blocking issue; it is explicitly documented as a V1 milestone in
`docs/roadmap.md` and `docs/kubernetes-validation.md`. Do not interpret the port 8472
support or scenario results as proof of flannel compatibility.

**No cloud deployment testing.** The fragmentation-observed verdict was confirmed in
a netns lab. Whether ip_do_fragment fires during actual cloud VXLAN traffic, and
whether cloud fabric silently drops the resulting fragments, has not been tested.

**ARM64 live demo not run.** The end-to-end demo was executed from the amd64 packaged
archive only. The arm64 archive passed all structural and scenario gates in CI; a live
demo from the arm64 archive was not run.

## v0.1.0 vs V1

v0.1.0-rc1 is a lab-validated prototype. It is not production-ready and makes no
claim of production correctness, all-kernel compatibility, or Kubernetes validation.

V1 targets (see `docs/roadmap.md`): Kubernetes/CNI validation, per-VNI attribution,
rolling-window comparison for PTB suppression, additional kernel coverage.
